# 导航栏工作流单阶段脉冲提示设计规格 (Spec)

## 1. 目标与背景

当前导航栏中的工作流组件会展示一条分段式阶段进度条和 `当前阶段/总阶段数`。这虽然能完整表达整个工作流的推进情况，但也会把“后面还有很多阶段”直接暴露给用户。对于希望降低心理负担、只专注于当前这一段的用户，这种表达过于强调总量。

本次设计的目标是：

1. 将导航栏工作流展示改为“只突出当前阶段”的心智模型。
2. 保留当前阶段、计时信息和打开工作流详情的入口价值。
3. 去掉会强化“剩余流程很多”的分段条和总阶段数提示。
4. 用极弱的动态脉冲保留一点连续节奏感，但不允许用户把它理解成剩余阶段数量。

## 2. 非目标

本次不做以下事情：

1. 不改动工作流后端接口、阶段推进逻辑或任何数据库结构。
2. 不改变导航栏外的工作流展示，例如卡片区的完整 `WorkflowProgressBar`。
3. 不新增新的工作流状态类型，只基于现有 `focus` / `break` / `pending_task_selection` / `pending_confirmation` 等状态重新组织 UI。
4. 不在导航栏中补充总进度百分比、剩余阶段数或下一阶段列表。

## 3. 核心设计总览

### 3.1 总体方向

导航栏中间的工作流组件继续保持“可点击的紧凑胶囊”形态，但从“全流程摘要”改为“当前阶段状态卡”。

信息优先级调整为：

1. 当前阶段类型
2. 当前计时信息
3. 极弱的节奏提示

不再展示：

1. 分段式进度条
2. `1/4`、`2/5` 之类的阶段序号
3. 可以推断剩余流程总量的视觉分格

### 3.2 选定视觉方向

采用用户确认的 `B + 1` 方向，并进一步收敛到“轻脉冲”变体：

1. 主体仍然是阶段胶囊。
2. 右侧时间数字左边加入一组轻量脉冲。
3. 该脉冲组只传达“流程节奏仍在延续”，不表达步骤数量。
4. 脉冲的视觉语义更接近呼吸或节拍，而不是 loading 或步骤指示器。

## 5. 视觉与交互设计

### 5.1 基础结构

导航栏胶囊从左到右的结构为：

1. 当前状态点
2. 阶段文案区
3. 轻脉冲组
4. 计时区或动作区

其中：

1. 状态点保留现有颜色语义。
2. 阶段文案区承担“现在在做什么”的主要认知入口。
3. 轻脉冲组位于时间左侧，作为弱节奏提示。
4. 右侧仍保留时间或操作按钮，方便快速扫视。

### 5.2 颜色语义

1. `focus` 阶段使用蓝色系。
2. `break` 阶段使用橙色系。
3. `pending_task_selection` 与 `pending_confirmation` 进入静态模式，可沿用当前阶段主色，但不显示脉冲。

颜色切换只改变强调色，不改变布局结构，避免用户在专注态和休息态之间重新学习组件。

### 5.3 轻脉冲组定义

轻脉冲组采用三个元素组成的非计数化表达：

1. 一个较亮、较宽的主脉冲
2. 两个更淡、更短的尾迹

约束如下：

1. 三个元素不能做成等宽等距的“步骤点”，避免被理解为剩余阶段数。
2. 主脉冲应明显亮于尾迹，但整体存在感要弱于时间数字。
3. 脉冲动画以明暗和轻微宽度变化为主，不做明显位移。
4. 动画节奏建议控制在 `1.2s` 到 `1.6s` 一个循环。
5. 动画应保持低对比度，避免形成“页面正在加载”的误解。

### 5.4 去除的元素

以下内容从导航栏组件中移除：

1. 横向分段进度条容器
2. 每个阶段对应的 fill 计算与渲染
3. 右侧 `activePhaseIndex + 1 / phases.length` 文案
4. 任意可以感知“总共有多少段”的视觉元素

## 6. 状态设计

### 6.1 进行中的专注阶段

展示要求：

1. 主文案显示 `专注阶段`
2. 右侧显示当前计时信息
3. 显示蓝色轻脉冲组
4. 若为 `countup`，可继续显示极弱的目标提示，例如 `目标 25:00`

### 6.2 进行中的休息阶段

展示要求：

1. 主文案显示 `休息阶段`
2. 右侧显示剩余时间
3. 显示橙色轻脉冲组
4. 布局不变，只切换强调色

### 6.3 待选任务状态

展示要求：

1. 主文案改为 `选择任务`
2. 不显示轻脉冲组
3. 右侧不显示运行中的时间动画语义，可保留静态说明或现有时间占位逻辑
4. 整体视觉明确区别于“阶段正在进行中”

### 6.4 待确认继续状态

展示要求：

1. 保留当前 `继续` 按钮的交互价值
2. 不显示轻脉冲组
3. 保持它是一个明确的动作态，而不是运行态

### 6.5 正常态

当 `workflow.state === 'normal'` 或缺少有效阶段数据时，组件继续返回 `null`，不在导航栏中显示任何内容。

## 7. 文案与可访问性

### 7.1 文案原则

导航栏只承载当前状态，不承载流程总量说明。

因此：

1. 主文案只说当前阶段，例如 `专注阶段`、`休息阶段`、`选择任务`
2. 不出现 `第 2 阶段`、`共 5 阶段` 等文案
3. 辅助文案如果存在，也只描述当前动作语境，不描述剩余流程

### 7.2 可访问性要求

1. 保持整个胶囊可通过 `Enter` / `Space` 打开工作流弹窗。
2. `aria-label` 需要同步更新，描述当前阶段和时间，但不再提及总阶段数。
3. 若显示按钮（例如 `继续`），其点击应继续阻止冒泡，避免误触整块胶囊的详情打开行为。
4. 动画需要尊重低干扰原则；若后续项目已有 `prefers-reduced-motion` 约定，实现时应兼容。

## 9. 测试策略

至少补充组件层测试，覆盖以下行为：

1. 进行中的 `focus` 状态显示轻脉冲组，不显示旧分段条，不显示阶段总数。
2. 进行中的 `break` 状态也显示轻脉冲组，但使用休息阶段语义。
3. `pending_task_selection` 状态隐藏轻脉冲组。
4. `pending_confirmation` 状态隐藏轻脉冲组，并保留 `继续` 操作。
5. `countup` 状态仍然正确显示累计时间和目标提示。

测试应优先放在现有前端测试目录中，避免只依赖手工视觉回归。

## 10. 风险与取舍

1. 去掉总进度后，导航栏的“流程完成度”信息会明显变弱，但这是有意取舍，目标就是降低总量感知。
2. 轻脉冲如果做得过强，容易被误读成 loading；如果做得过弱，又可能退化成普通计时胶囊。实现时需要先以克制版本为默认。
3. 如果后续用户仍希望查看完整工作流结构，应该通过现有详情弹窗承载，而不是重新把总量信息塞回导航栏。
