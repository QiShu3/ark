# 事件成就板块设计

## 概述

本设计为首页右侧“成就”入口补全一个轻量但正式的成就系统。它采用用户已确认的“徽章 / 解锁感”视觉方向，并明确拆成两个 UI 状态：

- `卡片状态`：首页右侧的小卡片，只负责摘要和入口
- `弹窗状态`：点击卡片后打开的大弹窗，负责完整浏览、事件切换和成就明细

成就分成两条并行主线：

- `事件成就`：围绕某一个事件单独计算，切换事件就切换该事件的成就进度
- `全局成就`：跨所有事件长期累计，始终显示，不因当前事件存在而隐藏

这个方向与当前仓库里的事件模型一致。`event` 已经不是松散标签，而是任务和日程的正式锚点对象，适合作为“这一阶段正在推进什么目标”的成就上下文。

## 已确认的产品决策

在本轮讨论中，用户明确确认了以下产品规则：

- 成就板块采用更有“解锁感 / 徽章感”的方向，而不是成长值面板优先
- 点击后的第一屏优先展示“最近解锁”
- 第一版成就来源采用均衡混合：专注、任务完成、日程确认都参与
- 每个事件拥有自己独立的一套成就进度
- 切换事件时，弹窗中的事件成就区域随之切换
- 全局成就始终存在，即使当前存在主事件也会显示
- 成就板块必须明确区分两个 UI：`卡片状态` 与 `弹窗状态`
- “战绩”不作为正式产品语言，统一使用 `事件成就`

## 目标

- 为首页右侧现有“成就”占位入口提供真实可用的产品形态
- 让事件成为成就系统的自然上下文，而不是孤立的倒计时对象
- 在不大改现有任务 / 日程 / 事件模型的前提下，先做出可读、可切换、可扩展的成就系统
- 保持全局成就始终可见，避免用户只能在特定页签中看到长期累计
- 将复杂计算收敛到后端，避免前端重复推导规则

## 非目标

- 第一版不做完整的“成就历史馆 / 纪念册”
- 第一版不做复杂的等级树、技能树、排行或社交玩法
- 第一版不引入动画很重的解锁庆祝系统
- 第一版不要求记录每个成就的永久解锁时间线
- 第一版不改造事件自身的编辑入口结构，只消费现有事件模型

## 当前上下文

当前仓库已经提供了成就系统可依赖的基础对象：

- `events`：正式事件表，包含 `name`、`due_at`、`is_primary`
- `tasks.event_id`：任务可以正式绑定一个事件
- `appointments.event_id`：日程可以正式绑定一个事件
- `time_inherits_from_event` / `time_overridden`：安排时间可以继承事件时间
- `focus_logs`：已有专注记录，可通过任务回溯到所属事件
- `completion_records`：已有任务 / 日程的完成记录
- `EventCountdownCard`：首页左侧已有主事件展示入口

这说明产品上已经存在“当前主事件”概念，成就板块应复用而不是重新发明上下文。

## 概念模型

### 1. 事件成就

`事件成就` 属于某一个具体事件，例如“论文投稿”或“期末考试”。

它只统计与该 `event_id` 直接相关的行为：

- 绑定到该事件的任务完成
- 绑定到该事件的日程确认 / 完成
- 与这些绑定任务相关的专注记录

它不统计未绑定事件的行为，也不统计其他事件的行为。

### 2. 全局成就

`全局成就` 是跨事件长期累计的成就层。

它统计所有行为：

- 包含绑定到事件的任务 / 日程
- 也包含未绑定事件的任务 / 日程
- 包含全部专注记录

全局成就不是一个独立页面，也不是只有在没有主事件时才显示。它应该在卡片和弹窗中都作为常驻层存在。

### 3. 当前主事件

首页卡片默认跟随当前 `主事件`。

- 如果存在主事件，卡片显示该主事件的事件成就摘要
- 如果不存在主事件，卡片进入“暂无主事件成就”的空状态
- 无论是否存在主事件，全局成就摘要都应显示

## UI 结构

### 卡片状态

### 目的

卡片状态的目的不是完整浏览成就，而是回答三个问题：

1. 当前主事件最近有什么成就进展
2. 下一个最接近的事件成就是什么
3. 全局长期积累有没有继续推进

### 内容结构

卡片应包含以下信息：

- 标题：`成就`
- 当前主事件名称，例如 `论文投稿`
- 一条事件成就主摘要，例如：
  - `事件已解锁 3 项 · 最近「连续推进 3 天」`
- 一条下一步提示，例如：
  - `再完成 1 个绑定安排，解锁「收尾干净」`
- 一条全局成就摘要，例如：
  - `全局：总专注 42h · 距「专注 50h」还差 8h`

### 空状态

没有主事件时：

- 事件成就区域显示 `暂无主事件成就`
- 提示用户先设置主事件，或先推进全局成就
- 全局成就摘要仍然照常显示

没有任何已解锁成就时：

- 卡片不显示空白
- 改为显示第一个最接近的可达成目标，例如：
  - `完成 1 个绑定安排即可解锁首个事件徽章`

### 交互

- 卡片整体可点击
- 点击后打开成就弹窗
- 卡片本身不提供事件切换交互

### 弹窗状态

### 目的

弹窗状态负责完整浏览：

- 查看当前事件成就详情
- 切换不同事件
- 查看全局成就
- 浏览最近解锁和即将解锁

### 布局结构

弹窗分为上下两大区域：

#### 上半部分：当前事件成就

- 顶部标题，例如 `成就 · 论文投稿`
- 事件切换 tabs
- 当前事件统计摘要
- 最近解锁
- 即将解锁

这一部分随着事件切换而变化。

#### 下半部分：全局成就

- 全局统计摘要
- 全局成就卡片列表

这一部分始终显示，不随事件切换消失。

### 为什么不使用“全局”页签

用户已经明确确认：全局成就总是出现，即使存在事件时也会显示全局成就。

因此第一版不应设计为：

- `论文投稿 | 期末考试 | 全局`

而应设计为：

- 顶部仅切换事件
- 当前事件成就在上
- 全局成就在下

这样符合“事件是当前上下文，全局是长期底座”的产品语义。

## 成就计算规则

### 第一版总体策略

第一版采用 `派生计算`，不强制新增“成就解锁历史表”。

也就是说，后端基于现有数据源即时计算每个成就的状态：

- `unlocked`
- `in_progress`
- `locked`

并返回：

- 当前值
- 目标值
- 是否已解锁
- 简短说明

这样可以在不引入额外持久化复杂度的前提下先验证玩法和展示结构。

后续如果需要以下能力，再新增持久化解锁记录：

- 首次解锁时间
- 解锁时间线
- 成就通知中心
- 成就纪念册

## 事件成就的数据来源

事件成就只统计绑定到当前 `event_id` 的行为。

### 任务

任务满足以下条件时计入事件成就：

- `tasks.event_id == current_event_id`

可用指标：

- 完成任务数
- 连续推进天数
- 截止前完成关键任务

### 日程

日程满足以下条件时计入事件成就：

- `appointments.event_id == current_event_id`

可用指标：

- 已确认 / 已出席次数
- 关键节点确认次数

### 专注

专注记录本身没有 `event_id`，需要通过 `focus_logs.task_id -> tasks.event_id` 回溯：

- 如果该专注记录关联的任务绑定到当前事件，则记入该事件专注时长
- 如果该任务没有事件绑定，则不记入任何事件成就
- 但它仍然记入全局成就

## 全局成就的数据来源

全局成就统计所有行为，不需要 `event_id`。

可直接基于：

- `focus_logs`
- `completion_records`
- 任务 / 日程总体完成数
- 使用天数

## 第一版成就池

第一版保持成就池有明确边界。

### 事件成就

推荐先做 6 个：

1. `首次推进`
   - 第一次完成绑定到该事件的任务或日程
2. `连续推进 3 天`
   - 连续三天至少完成一项绑定该事件的行为
3. `进入状态`
   - 围绕该事件累计专注达到 2 小时
4. `推进不停`
   - 完成 5 个绑定安排
5. `守住关键节点`
   - 确认 3 个绑定日程
6. `交付前夜`
   - 在事件截止前 24 小时内完成一个此前未完成的绑定安排，且完成后该事件不再剩余未完成的绑定安排

### 全局成就

推荐先做 4 个：

1. `总专注 10 小时`
2. `总专注 50 小时`
3. `累计完成 100 项安排`
4. `连续使用 7 天`

## 连续推进的定义

第一版需要一个明确且可实现的定义：

- 某天只要发生至少一个“绑定到该事件的有效完成行为”，即视为这一天有推进
- 有效完成行为包括：
  - 绑定任务的完成记录
  - 绑定日程的确认 / 完成记录
- 是否把“仅有专注但未完成任何安排”算作推进，第一版建议暂不计入

这样定义更稳定，也更容易向用户解释。

## 接口设计

建议新增一个轻量汇总接口：

`GET /todo/achievements/summary?event_id=<optional>`

行为规则：

- 传入 `event_id` 时，返回该事件的事件成就 + 全局成就
- 不传时，默认返回当前主事件的事件成就 + 全局成就
- 没有主事件时，`event_achievements` 返回 `null`
- 无论是否存在主事件，`global_achievements` 都必须返回

### 推荐响应结构

```ts
type AchievementStatus = 'unlocked' | 'in_progress' | 'locked';

type AchievementItem = {
  id: string;
  title: string;
  description: string;
  status: AchievementStatus;
  current_value: number | null;
  target_value: number | null;
  progress_text: string | null;
};

type AchievementSection = {
  title: string;
  summary_text: string | null;
  stats: {
    unlocked_count: number;
    in_progress_count: number;
    primary_metric_value: number | null;
    primary_metric_label: string | null;
  };
  latest_unlocked: AchievementItem[];
  upcoming: AchievementItem[];
};

type AchievementSummaryResponse = {
  active_event: {
    id: string;
    name: string;
    due_at: string;
    is_primary: boolean;
  } | null;
  event_achievements: AchievementSection | null;
  global_achievements: AchievementSection;
};
```

### 为什么不用多个散接口

第一版不建议拆成：

- `/event-achievements`
- `/global-achievements`
- `/achievement-progress`

因为首页卡片和弹窗都会同时消费事件与全局信息。一个汇总接口可以减少前端拼装复杂度，也更容易统一缓存和空状态。

## 前端组件拆分

第一版不应继续把逻辑堆进 `PlaceholderCard.tsx`。

推荐拆分为：

- `AchievementCard.tsx`
- `AchievementModal.tsx`
- `AchievementBadgeCard.tsx`
- `achievementTypes.ts`

### AchievementCard

负责：

- 请求汇总接口
- 渲染卡片状态
- 打开 / 关闭弹窗

不负责：

- 事件切换逻辑
- 大量列表渲染细节

### AchievementModal

负责：

- 请求事件列表 `/todo/events`
- 切换当前事件
- 请求指定事件的成就汇总
- 渲染上半部分事件成就和下半部分全局成就

### AchievementBadgeCard

负责渲染单个成就卡片：

- 徽章图标
- 标题
- 描述
- 已解锁 / 进行中 / 未解锁状态

### PlaceholderCard 的职责

`PlaceholderCard.tsx` 在 `index === 3 && subIndex === 0` 时只渲染 `AchievementCard`，不再承载成就规则和展示逻辑。

## 数据流

### 首页卡片

1. 页面加载
2. `AchievementCard` 请求 `/todo/achievements/summary`
3. 后端默认按当前主事件返回 `event_achievements + global_achievements`
4. 卡片展示主事件摘要和全局摘要

### 打开弹窗

1. 用户点击卡片
2. `AchievementModal` 打开
3. 加载 `/todo/events`
4. 事件 tabs 展示全部事件
5. 默认选中当前主事件
6. 上半部分显示当前事件成就
7. 下半部分显示全局成就

### 切换事件

1. 用户点击某个事件 tab
2. 前端请求 `/todo/achievements/summary?event_id=<selected>`
3. 更新弹窗上半部分
4. 全局成就区域保持不变，或者复用同一响应中的 `global_achievements`

## 空状态与错误处理

## 没有主事件

卡片：

- 显示 `暂无主事件成就`
- 同时显示全局成就摘要
- 提示“设置主事件后可查看事件成就”

弹窗：

- 事件 tabs 仍显示全部事件
- 如果用户还没有创建任何事件，则事件成就区域显示空状态说明
- 全局成就区域仍正常渲染

## 没有任何事件

- 事件成就区域显示空状态引导
- 不隐藏全局成就

## 接口失败

卡片：

- 使用温和降级文案，例如 `成就加载失败，可稍后重试`
- 尽量保留卡片壳子，不直接消失

弹窗：

- 显示错误提示和重试入口
- 如果事件列表加载成功但成就汇总失败，可仍保留事件 tabs

## 测试

### 前端测试

应至少覆盖：

- 首页成就卡渲染主事件摘要
- 首页成就卡在没有主事件时仍显示全局成就摘要
- 点击卡片打开弹窗
- 弹窗加载事件列表
- 切换事件时仅事件成就区域变化
- 全局成就区域在切换事件后仍然可见
- 空状态与错误状态正确展示

### 后端测试

应至少覆盖：

- 默认返回主事件的事件成就
- 指定 `event_id` 时返回对应事件成就
- 不存在主事件时 `event_achievements == null`
- `global_achievements` 在所有情况下都返回
- 事件成就只统计绑定到该 `event_id` 的任务 / 日程 / 专注
- 全局成就统计所有行为
- 专注通过 `task_id -> event_id` 正确归属到事件

## 实施顺序

### Phase 1

- 新增后端汇总接口
- 新增前端成就卡与弹窗组件
- 接入事件切换
- 实现第一版事件成就 6 个和全局成就 4 个

### Phase 2

- 为成就补充首次解锁时间
- 增加更丰富的“最近解锁”排序能力
- 优化卡片和弹窗中的视觉节奏

### Phase 3

- 成就解锁通知
- 成就历史馆 / 纪念册
- 更复杂的事件阶段成就

## 设计结论

第一版成就系统的核心不是做一个泛化等级面板，而是围绕现有 `event` 模型建立“当前事件成就 + 常驻全局成就”双层结构。

首页卡片只做摘要和入口，弹窗负责完整浏览。事件切换只影响事件成就区域；全局成就始终存在。这样既符合用户确认的产品方向，也最大程度复用了当前仓库已经存在的事件、安排、专注与完成记录模型。
