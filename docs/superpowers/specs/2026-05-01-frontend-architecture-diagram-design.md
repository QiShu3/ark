# 前端架构图设计

## 背景

当前 Ark 前端已经不再是单一页面，而是由统一应用外壳承载多个业务入口：

- 首页工作台
- 应用中心
- 智能体会话台
- 论文阅读台

对于新成员而言，直接阅读代码会很快撞到两个问题：

1. 先看不出系统有哪些前端入口，以及它们共享什么底座
2. 即使找到了首页或会话页，也不容易第一眼判断复杂度主要集中在哪些区域

因此需要一张面向开发协作的前端架构图，优先服务“新成员快速理解项目”，而不是展示所有实现细节。

## 目标

- 帮助新成员在短时间内建立 Ark 前端的整体地图
- 让读图者先理解业务区域，再顺着图回到代码
- 明确统一应用外壳、主要页面工作区和共享能力之间的关系
- 对首页工作台和智能体会话台做局部展开，覆盖最值得理解的内部结构

## 非目标

- 不做完整组件索引图
- 不逐一展示所有弹窗、子组件和测试文件
- 不在主图里罗列所有接口 endpoint
- 不将图做成纯实现导向的文件依赖图

## 受众与使用场景

主要受众：

- 新加入项目的前端开发者
- 需要快速理解前端分区的全栈开发者
- 参与讨论但暂时不深入代码细节的协作者

主要场景：

- 新成员 onboarding
- 需求评审时快速解释前端结构
- 后续补画更细分专题图之前的总览入口

## 命名原则

本图采用“业务词为主，技术词为辅”的命名策略。

### 主命名规则

- 优先使用业务名称表达图中节点含义
- 节点名称应让不熟悉代码的人也能直接理解用途

示例：

- `登录入口`
- `应用框架`
- `首页工作台`
- `应用中心`
- `智能体会话台`
- `论文阅读台`

### 辅助命名规则

- 对影响理解的技术机制，可在副标题或小字中补充
- 技术词不作为主标题，但可作为数据流或能力说明

示例：

- `会话与鉴权`
- `共享数据访问`
- `实时消息通道`
- `语音播报能力`
- `REST API`
- `WebSocket`

## 总体方案

最终交付采用：

- `1 张主图`
- `2 个局部展开`

即：

1. 前端整体主架构图
2. 首页工作台局部展开图
3. 智能体会话台关键链路展开图

这样既能保持主图清晰，又能覆盖最值得理解的复杂区域。

## 方案对比

在讨论中评估过三种画法：

### 方案一：路由优先

按“路由 -> 页面”组织主图。

优点：

- 最容易快速说明系统有哪些页面
- 适合第一次介绍项目入口

缺点：

- 不够突出共享能力层
- 不容易表达首页和会话页的内部结构

### 方案二：分层优先

按“页面层 -> 组件层 -> hooks/lib 层”组织主图。

优点：

- 适合表达前端架构分层
- 技术结构比较清楚

缺点：

- 新人第一次看时，不容易建立页面地图
- 容易变成抽象分层图，而不是实际工作入口图

### 方案三：混合级主图

先画应用外壳与页面地图，再对关键页面做局部展开。

优点：

- 同时兼顾可读性和开发价值
- 更符合本项目“多个业务页共享统一外壳”的实际结构
- 适合新成员先建立全局认知，再进入重点区域

缺点：

- 需要控制信息密度，否则主图和局部图容易重复

## 选型结论

采用方案三：混合级主图。

原因：

- 最符合“新成员快速理解项目”的目标
- 能先回答“系统里有什么”，再回答“复杂度在哪里”
- 适合后续继续扩展更多专题子图，而不破坏主图结构

## 主图设计

主图采用三层结构，自上而下阅读：

1. `应用框架层`
2. `页面工作区层`
3. `共享能力层`

### 1. 应用框架层

这一层用于表达所有业务页面都运行在同一个统一外壳中。

建议包含的业务节点：

- `应用框架`
- `登录入口`
- `会话与鉴权`
- `顶部导航`
- `全局背景与页脚`

这一层要传达的信息：

- 登录后页面并不是各自独立存在，而是挂在统一应用壳子中
- 顶部导航、背景氛围和页脚属于跨页共享框架能力
- 页面切换和访问权限控制先经过统一入口，而不是各页各自处理

### 2. 页面工作区层

这一层用于建立页面地图，帮助新人快速理解“系统里有哪些前端入口”。

建议包含的业务节点：

- `首页工作台`
- `应用中心`
- `智能体会话台`
- `论文阅读台`

`登录入口` 保持在框架层附近，作为进入系统的前置路径，而不与工作区页面并列。

这一层要传达的信息：

- 首页工作台是主工作区，不等同于普通菜单页
- 应用中心承担“跳转到不同能力”的入口职责
- 智能体会话台和论文阅读台是独立业务区，而不是首页内部一个小面板

### 3. 共享能力层

这一层用于表达多个页面共同依赖的前端能力。

建议包含的业务节点：

- `身份状态管理`
- `共享数据访问`
- `实时消息通道`
- `语音播报能力`
- `工作流提醒与同步`
- `本地持久化`

可在图的边缘补充外部交互对象：

- `REST API`
- `WebSocket`

这一层要传达的信息：

- 多个页面不是各自直连后端的“散装页面”
- 会话、鉴权、请求封装、提醒和本地持久化已经形成共享底座
- 实时消息和语音能力是系统的重要特征，不是页面私有技巧

## 局部展开一：首页工作台

首页工作台适合画成“布局骨架 + 业务聚合区”的结构。

### 目标

- 解释首页工作台的阅读路径
- 让新人理解复杂度并不在外层容器，而主要集中在右侧功能聚合区

### 建议结构

- `首页工作台`
  - `沉浸主视区`
  - `右侧功能区`

其中：

- `沉浸主视区` 承载角色展示、事件倒计时、对话互动等偏场景化体验
- `右侧功能区` 承载多个具体生产力模块，是首页业务复杂度最集中的区域

### 右侧功能区建议展开内容

建议作为同级业务模块呈现：

- `专注流与进度`
- `任务与安排`
- `日历视图`
- `成就与反馈`
- `快捷创建与编辑`
- `移动端预览`

### 图上需要明确的结论

- 首页工作台不是普通 dashboard，而是“沉浸体验区 + 业务操作区”的组合
- 新人如果要改首页业务，优先应该理解右侧功能区，而不是停留在最外层容器
- 首页中很多弹窗和编辑动作都属于右侧聚合区的延伸能力

## 局部展开二：智能体会话台

智能体会话台不建议仅用组件树表示，而应画成“初始化 + 实时链路 + 呈现支线”的结构。

### 目标

- 解释这不是普通 CRUD 页面
- 强调其核心价值在于会话驱动和实时通信

### 建议结构

- `智能体会话台`
  - `会话初始化`
  - `实时消息通道`
  - `消息呈现`
  - `语音播报支线`

### 关键链路说明

#### 会话初始化

- 恢复或创建当前页面对应的会话
- 拉取历史消息，建立初始上下文

#### 实时消息通道

- 页面进入持续连接状态
- 接收会话事件、生成状态变化与流式内容

#### 消息呈现

- 将历史消息和实时消息整合为当前对话
- 区分流式增量内容与最终消息结果

#### 语音播报支线

- 作为会话事件的附加消费链路存在
- 不喧宾夺主，但需要标出其依赖实时事件

### 图上需要明确的结论

- 智能体会话台采用“REST 初始化 + WebSocket 持续更新”的组合模式
- 会话对象是页面级核心锚点
- 语音播报能力依附于实时消息流，而不是独立页面流程

## 连接线语义

为避免主图过密，连线应只保留 3 种语义：

- `实线`：结构包含或承载关系
- `虚线`：共享依赖关系
- `高亮箭头`：关键数据流或实时链路

建议用法：

- 应用框架到各页面：实线
- 页面到共享能力层：虚线
- 智能体会话台到 `WebSocket`：高亮箭头

## 视觉组织建议

建议采用横向泳道或分层板块布局：

- 顶部：`应用框架层`
- 中部：`页面工作区层`
- 底部：`共享能力层`
- 右侧或下方：`首页工作台` 与 `智能体会话台` 的局部 callout

风格建议：

- 主图节点数量控制在易于扫读的范围内
- 业务层节点名称简洁明确
- 技术机制通过小字、副标题或细标签补充
- 局部展开区采用和主图一致的色彩体系，但视觉上稍弱于主图标题区

## 不建议放入图中的内容

以下内容会增加噪音，不建议进入首版架构图：

- 所有小组件的逐一列举
- 所有弹窗的逐一展开
- 所有接口路径的完整罗列
- 测试文件、故事文件、样式文件
- 纯视觉装饰或动画细节

## 输出建议

若进入正式出图阶段，建议交付以下产物：

- `frontend-architecture.drawio`
- `frontend-architecture.spec.yaml`
- `frontend-architecture.arch.json`
- `frontend-architecture.png`

若需要便于文档嵌入，也可补充：

- `frontend-architecture.svg`

## 验收标准

- 新成员读图后能说出系统的主要前端入口
- 能理解这些页面运行在统一应用框架之下
- 能明确首页工作台与智能体会话台为何需要单独展开
- 主图不会因细节过多而失去可读性
- 节点命名以业务词为主，不依赖读者先知道真实文件名

## 后续实施建议

1. 先按本设计完成首版 draw.io 主图和两个局部展开
2. 导出 PNG 进行一次视觉检查，确认层次、连线和标签密度是否合适
3. 如团队后续需要，再从主图延伸出更细的专题图，例如：
   - 首页工作台详细业务图
   - 智能体会话链路图
   - 论文阅读台数据流图
