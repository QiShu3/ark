# 导航栏工作流灵动岛化二次优化设计规格 (Spec)

## 1. 目标与背景

当前导航栏工作流组件已经从“分段总进度条”收敛到了“单阶段胶囊 + 轻脉冲”的表达，解决了总量感带来的心理压力问题。但这仍然是一个单层级状态组件：不同运行状态下的信息密度基本相同，不能像灵动岛那样在“平时克制”和“需要更多反馈时自然展开”之间切换。

本次设计的目标是：

1. 将导航栏工作流组件升级为更接近“灵动岛”体验的三层状态组件。
2. 让常驻态更短、更克制，只保留最必要信息。
3. 在用户主动靠近或阶段切换时，短暂提升信息密度与反馈感。
4. 始终保留时间信息，不让反馈或展开打断用户的时间锚点。

## 2. 非目标

本次不做以下事情：

1. 不改动工作流后端接口、阶段推进逻辑或工作流详情弹层的数据结构。
2. 不把导航栏组件做成独立浮层、toast 或系统通知样式。
3. 不在休息态、待选任务态或待确认态的 hover 中强行补充额外信息。
4. 不把完整工作流结构、后续阶段列表或阶段总数重新放回导航栏。

## 3. 核心设计总览

### 3.1 总体方向

导航栏工作流组件采用“灵动岛化”的三层状态模型：

1. 常驻态
2. 悬浮展开态
3. 切换反馈态

其核心原则是：

1. 平时尽量短，减少存在感。
2. 用户主动关注时，才暴露更具体的信息。
3. 阶段刚切换时，给一个短促、明确但不打断的反馈。

### 3.2 三层状态定义

#### 常驻态

常驻态是导航栏里最常见的形态。

特征：

1. 胶囊长度较短。
2. 仅显示当前状态、时间和极弱节奏提示。
3. 不显示任务名。
4. 不显示副文案。

示例：

- `专注中   12:34`
- `休息中   04:56`

#### 悬浮展开态

悬浮展开态只在 `专注中` 时出现。

特征：

1. 用户鼠标悬浮在胶囊上时触发。
2. 胶囊平滑变长。
3. 展示“当前专注任务”。
4. 鼠标移出后自动恢复为短常驻态。
5. 时间仍保留在右侧。

这是一个用户主动拉取更多信息的层级，而不是系统主动打断。

#### 切换反馈态

切换反馈态发生在阶段切换瞬间。

特征：

1. 当 `focus -> break` 或 `break -> focus` 等切换发生时，胶囊短暂变长。
2. 中间文案切换为反馈文案，例如 `专注完成`、`开始休息`、`休息结束`、`继续专注`。
3. 时间继续显示，不被反馈文案替代。
4. 动效结束后，自动回到对应的新阶段常驻态。

这一层承担“被看见的进展”与“继续往下走”的情绪价值。

## 4. 状态与长度设计

### 4.1 长度策略

组件长度不再固定，而是跟状态层级相关：

1. 常驻态最短
2. 悬浮展开态中等长度
3. 切换反馈态最长

设计原则：

1. 长度变化必须可感知，但不能夸张到像弹窗或浮层。
2. 变长只发生在有额外信息或反馈时，不是常态。
3. 收缩回常驻态的过程要自然，不能突兀消失。

### 4.2 各状态长度建议

相对关系建议如下：

1. 常驻态：只够容纳状态文案、轻脉冲和时间。
2. 悬浮展开态：在常驻态基础上增加任务名空间。
3. 切换反馈态：比 hover 态略长或相近，以容纳反馈文案和更醒一点的节奏感。

不要求三种状态绝对宽度固定一致，但要求：

1. 常驻态始终明显短于另外两种扩展态。
2. hover 展开和反馈展开都保持同一视觉骨架，避免像两个完全不同组件。

## 5. 各状态具体表现

### 5.1 专注中常驻态

展示内容：

1. 状态点
2. `专注中`
3. 极轻脉冲
4. 当前时间

不展示：

1. 当前任务名
2. 副文案
3. 任何流程总量信息

### 5.2 专注中悬浮展开态

触发条件：

1. 当前状态为 `focus`
2. 胶囊被鼠标 hover
3. 存在当前任务标题

展示内容：

1. 状态点
2. `专注中`
3. 当前专注任务标题
4. 极轻脉冲
5. 当前时间

任务标题展示原则：

1. 它是悬浮态新增的核心信息。
2. 应该优先展示任务标题本身，而不是额外描述性文案。
3. 若标题过长，应采用单行截断，不挤掉时间。

### 5.3 休息中常驻态

展示内容：

1. 状态点
2. `休息中`
3. 暖色系轻脉冲
4. 当前时间

休息态不需要 hover 展开额外信息，因为当前没有“当前专注任务”这一层用户价值。

### 5.4 切换反馈态

建议反馈文案：

1. `专注完成`
2. `开始休息`
3. `休息结束`
4. `继续专注`
5. `已选择任务`
6. `可以继续了`

其中优先级最高的是前四个阶段切换文案。

反馈态特征：

1. 中间文案从状态文案切到反馈文案。
2. 胶囊长度临时变长。
3. 轻脉冲强度略提升一次。
4. 时间保持显示。

### 5.5 阻塞态

阻塞态包括：

1. `选择任务`
2. `等待继续`

规则：

1. 可以比常驻态略长，但不应拥有运行态的流动感。
2. 不显示轻脉冲。
3. 若带按钮（例如继续），长度以容纳按钮和时间为主，不追求反馈态那种“醒一下”的感觉。

## 6. 动效与交互原则

### 6.1 常驻态动效

常驻态应保持轻呼吸感：

1. 动效频率低
2. 对比度低
3. 存在感弱于时间信息

它更像“状态活着”，而不是“系统正在加载”。

### 6.2 悬浮展开动效

hover 动效应表现为：

1. 胶囊宽度平滑扩展
2. 任务名淡入或轻滑入
3. 时间位置保持稳定，不横向跳动过大

hover 结束时：

1. 任务名收起
2. 胶囊缩回常驻长度

### 6.3 切换反馈动效

切换反馈态应表现为：

1. 胶囊短暂扩展
2. 反馈文案替换中间主文案
3. 轻脉冲进行一次更明显的“重置”
4. 约 `1.2s - 1.8s` 后恢复常驻态

### 6.4 时间信息约束

时间是高优先级信息，任何状态都不能隐藏。

因此：

1. hover 展开时保留时间
2. 切换反馈时保留时间
3. 阻塞态也保留时间或零点状态

### 7.2 推荐的前端状态拆分

为避免组件逻辑变乱，推荐在组件内部显式区分：

1. 运行状态
2. hover 展开状态
3. 切换反馈状态

实现上不要把“是否正在 feedback”混在普通渲染分支里用临时判断堆出来，而是明确管理：

1. 当前基础阶段状态是什么
2. 当前是否处于短暂反馈窗口
3. 当前是否允许 hover 展开

### 7.3 与业务数据的关系

本次设计不新增后端字段，但需要继续利用已有数据：

1. 当前阶段类型
2. 当前任务标题
3. 当前时间
4. `pending_task_selection`
5. `pending_confirmation`

若当前没有 `task_title`，即使处于专注态，hover 也不应为了展开而展开出空白内容。

## 8. 测试策略

后续实现时，至少应覆盖以下场景：

1. `focus` 常驻态默认长度较短，不显示任务名。
2. `focus` hover 时胶囊变长并显示当前任务标题。
3. `focus` hover 结束后恢复短胶囊。
4. `break` 状态 hover 不展开任务信息。
5. 阶段切换反馈出现时，胶囊临时变长且时间仍然显示。
6. 反馈结束后恢复到新阶段常驻态。
7. `pending_task_selection` 和 `pending_confirmation` 不显示运行态脉冲。

## 9. 风险与取舍

1. 长度变化如果做得过大，导航栏会显得跳跃；如果做得太小，又达不到“灵动岛式层级感”。
2. hover 展开只对桌面端成立，未来如果需要兼容触屏设备，应另行定义触发方式。
3. 切换反馈态和 hover 展开态可能同时发生，后续实现时需要明确优先级。建议反馈态优先于 hover 态。
4. 如果任务标题过长且时间不能隐藏，布局上必须优先保证时间稳定，其次才是任务名完整性。

## 10. 推荐结论

本次二次优化推荐采用以下策略组合：

1. 常驻态使用短胶囊，只保留 `状态 + 时间 + 极轻节奏`
2. 仅在 `专注中` 支持 hover 展开显示当前任务名
3. 阶段切换时使用短暂反馈展开态
4. 所有状态都保留时间信息

这个组合最符合当前目标：减少压力、增强陪伴感、提升切换反馈，同时不让导航栏重新变成高信息密度区域。
