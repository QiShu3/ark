# 任务专注与工作流阶段任务绑定设计规格 (Spec)

## 1. 目标与背景

当前“任务专注”与“工作流”已经部分耦合：用户点击开始专注时，后端会直接创建一个活动工作流并进入 `focus` 阶段，而不是开启一个独立的裸计时器。现状存在两个明显限制：

1. 工作流只在整体层级绑定一个 `task_id`，无法为每个 `focus` 阶段绑定不同任务。
2. `focus` 阶段只支持倒计时，不支持带上限保护的正计时。

本次设计的目标是：

1. 让每个 `focus` 阶段支持可选绑定一个未完成任务。
2. 支持工作流级默认计时模式，并允许每个 `focus` 阶段覆盖为 `countdown` 或 `countup`。
3. 让 `countup` 阶段保留目标时长，同时在达到 `2h` 时自动结束，避免异常累计专注时长。
4. 在任务完成时，自动推进当前工作流，并跳过后续绑定到已完成任务的无效 `focus` 阶段及其失去意义的 `break` 阶段。
5. 对未绑定任务的 `focus` 阶段，要求用户在运行时临时选择任务后才能开始。

## 2. 非目标

本次不做以下事情：

1. 不重建工作流数据库主模型，不新增独立的“阶段实例表”。
2. 不引入通用工作流引擎或任意节点类型，阶段类型仍然限定为 `focus` / `break`。
3. 不实现无限正计时；`countup` 必须受 `2h` 硬上限保护。
4. 不支持绑定已完成任务，也不保留“任务完成后仍继续绑定”的历史状态。
5. 不在本次引入多任务并行专注或一个阶段同时绑定多个任务。

## 3. 核心设计总览

### 3.1 总体方向

采用“方案 A”：继续沿用现有 `focus_workflow_presets.phases` 与 `focus_workflows.phases` 的 JSON 结构，扩展每个阶段对象的字段，不引入新的阶段实例表。

活动工作流仍然保持：

1. 一条 `focus_workflows` 活动记录。
2. 一个 `current_phase_index` 指向当前阶段。
3. 一个运行中的 `focus_log` 作为当前专注记账载体。

### 3.2 设计原则

1. 预设配置与运行态状态分离：预设上的阶段绑定不应被运行时临时选任务污染。
2. 任务有效性前置判断：任何 `focus` 阶段开始前都必须验证绑定任务仍未完成。
3. 跳过规则集中化：不要把“跳过无效阶段”的判断散落在多个接口里。
4. 现有接口尽量延续，只在必要处补充字段和运行态分支。

## 4. 数据模型设计

### 4.1 阶段 JSON 扩展

当前阶段对象形状为：

```json
{
  "phase_type": "focus",
  "duration": 1500
}
```

调整为：

```json
{
  "phase_type": "focus",
  "duration": 1500,
  "timer_mode": "countdown",
  "task_id": "00000000-0000-0000-0000-000000000000"
}
```

字段约束：

1. `phase_type`
   - 仍然只能是 `focus` 或 `break`。
2. `duration`
   - 仍然是秒数。
   - 对 `countdown` 表示倒计时目标时长。
   - 对 `countup` 表示参考目标时长，而不是强制结束时间。
3. `timer_mode`
   - 仅对 `focus` 阶段有意义。
   - 可选值：`countdown`、`countup`。
   - 对 `break` 阶段忽略该字段，运行时一律按倒计时处理。
4. `task_id`
   - 仅对 `focus` 阶段有意义。
   - 可为空。
   - 若存在，必须引用当前用户下未删除、未完成的任务。

### 4.2 工作流预设新增字段

在 `focus_workflow_presets` 及其 API 模型中新增：

```text
default_focus_timer_mode: "countdown" | "countup"
```

语义：

1. 作为 `focus` 阶段未显式填写 `timer_mode` 时的默认值。
2. `break` 阶段不受该字段影响。
3. 为兼容旧数据，缺失时默认视为 `countdown`。

### 4.3 活动工作流运行态新增字段

在 `focus_workflows` 运行态记录中新增两个字段：

```text
pending_task_selection: bool
runtime_task_id: UUID | null
```

语义：

1. `pending_task_selection`
   - 当前阶段为 `focus` 且没有可用任务时置为 `true`。
   - 该状态下阶段尚未真正开始，不应创建新的 `focus_log`。
2. `runtime_task_id`
   - 保存本次运行时为当前阶段临时选择的任务。
   - 仅对当前活动阶段有效。
   - 阶段推进到下一阶段时，需要清空。

### 4.4 当前阶段有效任务解析规则

当前阶段真正用于记账与展示的任务，按以下优先级决定：

1. `runtime_task_id`
2. 当前 phase 上的 `task_id`
3. 若均为空，则视为“当前阶段无任务”，进入等待选任务状态

## 5. 阶段流转与运行规则

### 5.1 进入 `focus` 阶段

进入任一 `focus` 阶段时，系统必须执行以下检查：

1. 解析当前阶段的有效任务。
2. 如果解析结果为空：
   - 设置 `pending_task_selection = true`
   - 不创建 `focus_log`
   - 返回前端，要求用户选择一个未完成任务
3. 如果解析出的任务已完成：
   - 当前阶段视为不可执行
   - 直接进入“向后扫描下一个可执行阶段”逻辑
4. 如果任务有效且未完成：
   - 清理 `pending_task_selection`
   - 根据阶段计时模式开始阶段
   - 创建新的 `focus_log`

### 5.2 `countdown` 阶段

`countdown` 阶段保持现有语义：

1. 展示剩余时间。
2. 到时进入待确认状态。
3. 用户确认后流转到下一可执行阶段。

与现状不同的点是：

1. 当前阶段记账任务来自“当前阶段有效任务”，而不是工作流级单一 `task_id`。
2. 若任务在阶段运行中被完成，则不等待倒计时自然结束，而是立即结束当前阶段并自动推进。

### 5.3 `countup` 阶段

`countup` 阶段语义如下：

1. 展示从 `0` 开始累加的已用时。
2. `duration` 保留为目标时长，用作阶段参考目标。
3. 到达 `duration` 时：
   - 阶段进入“已达目标”状态
   - 前端应给出视觉提示或操作提示
   - 阶段不自动结束
4. 如果用户继续停留，计时继续向上累加。
5. 当已用时达到 `7200` 秒时：
   - 自动结束当前阶段
   - 自动进入后续可执行阶段扫描流程

### 5.4 任务完成时的自动推进

如果某个绑定任务在工作流运行中被标记为 `done`：

1. 若该任务是当前 `focus` 阶段的有效任务：
   - 立即结束当前阶段
   - 关闭当前 `focus_log`
   - 将实际已过时长累计进任务 `actual_duration`
   - 自动寻找下一个可执行阶段
2. 若该任务绑定在后续尚未执行的 `focus` 阶段：
   - 无需立刻修改工作流索引
   - 但在后续推进时，该阶段必须被自动跳过
3. 同时，当前阶段及后续阶段上的 phase `task_id` / `runtime_task_id` 如果引用了已完成任务，返回给前端前应视为无效绑定，不再作为可执行任务来源

### 5.5 向后扫描下一个可执行阶段

活动工作流推进时，不再简单取 `current_phase_index + 1`，而是执行一次扫描：

1. 从当前索引后一个阶段开始向后遍历。
2. 若遇到绑定已完成任务的 `focus` 阶段：
   - 跳过该阶段。
3. 若某个 `break` 只服务于即将被跳过的无效 `focus` 阶段链路：
   - 一并跳过。
4. 若遇到未绑定任务的 `focus` 阶段：
   - 停在该阶段
   - 设置 `pending_task_selection = true`
   - 等待用户选任务
5. 若遇到绑定未完成任务的 `focus` 阶段：
   - 进入该阶段
   - 创建新的 `focus_log`
6. 若遍历到末尾仍无可执行阶段：
   - 结束整个工作流

### 5.6 `break` 阶段保留规则

`break` 阶段原则上仍然正常存在，但不应孤立保留在一串无效任务之间。

具体规则：

1. 正常从一个有效 `focus` 阶段流转出来的 `break` 阶段保留。
2. 如果后续 `focus` 阶段因绑定已完成任务而整体不可执行，则其前导或伴随的无意义 `break` 也一起跳过。
3. 结果上，用户看到的工作流只在“有效任务阶段”之间保留真正有意义的休息段。

## 6. API 与后端改造范围

### 6.1 预设相关接口

涉及文件：

- [`backend/routes/todo_routes.py`](/Users/qishu/Project/ark/backend/routes/todo_routes.py)

需要修改：

1. `FocusWorkflowPhase`
2. `FocusWorkflowPresetCreateRequest`
3. `FocusWorkflowPresetUpdateRequest`
4. `FocusWorkflowPresetOut`
5. `_normalize_workflow_phases`
6. `_row_to_focus_workflow_preset`
7. 预设新增、更新、默认值回填逻辑

校验要求：

1. `focus` 阶段允许 `task_id = null`。
2. `focus` 阶段若有 `task_id`，创建或保存预设时必须校验任务存在且未完成。
3. `break` 阶段上的 `task_id` 与 `timer_mode` 必须忽略或清空。

### 6.2 活动工作流接口

涉及接口：

1. `POST /todo/tasks/{task_id}/focus/start`
2. `POST /todo/focus/workflow/create`
3. `GET /todo/focus/workflow/current`
4. `POST /todo/focus/workflow/skip_phase`
5. `POST /todo/focus/workflow/confirm`
6. 新增一个“为当前阶段选择任务并开始”的接口

建议新增接口：

```text
POST /todo/focus/workflow/select-task
```

请求体：

```json
{
  "task_id": "uuid"
}
```

语义：

1. 仅当当前阶段是 `focus` 且 `pending_task_selection = true` 时可调用。
2. 任务必须未完成。
3. 调用成功后：
   - 写入 `runtime_task_id`
   - 清除 `pending_task_selection`
   - 正式开始当前阶段
   - 创建新的 `focus_log`

### 6.3 阶段推进核心函数

建议抽出统一的后端函数，命名为：

```python
_advance_focus_workflow_to_next_executable_phase(...)
```

职责：

1. 扫描下一个可执行阶段。
2. 跳过绑定已完成任务的无效阶段与无意义 `break`。
3. 处理“无任务需等待选择”的状态。
4. 在需要时创建新的 `focus_log`。
5. 在没有可执行阶段时结束整个工作流。

该函数应被以下路径复用：

1. 阶段确认
2. 任务完成后的自动推进
3. `countup` 达到 `2h` 上限后的自动推进
4. `skip_phase` 后的流转

## 7. 前端设计

### 7.1 工作流预设编辑界面

涉及文件：

- [`frontend/src/components/PlaceholderCard.tsx`](/Users/qishu/Project/ark/frontend/src/components/PlaceholderCard.tsx)

需要新增能力：

1. 工作流级默认计时模式选择器：
   - `倒计时`
   - `正计时`
2. 每个 `focus` 阶段支持：
   - 选择是否覆盖默认计时模式
   - 选择绑定任务（可为空）
3. 阶段任务选择器只展示未完成任务。
4. 若某个预设中绑定的任务后来完成：
   - 预设编辑界面中应显示为“已失效绑定”
   - 保存时自动清空或要求用户重新确认

### 7.2 运行中卡片与进度展示

1. 当前阶段若为 `countdown`：
   - 继续显示剩余时间。
2. 当前阶段若为 `countup`：
   - 显示已用时间。
   - 到目标时长后给出“已达到目标”提示。
   - 接近 `2h` 上限时建议有风险提示。
3. 当前阶段若 `pending_task_selection = true`：
   - 主卡片不应展示正常计时
   - 应展示“选择任务后开始当前阶段”的明显提示
   - 提供任务选择入口
4. 当前阶段绑定任务若在运行中完成：
   - 前端在收到刷新结果后应无缝切换到下一可执行阶段，或进入待选任务状态

### 7.3 任务选择交互

当前已有的 focus task picker 可作为基础，但需要升级语义：

1. 当前只是“切换即将专注的目标任务”。
2. 新设计里还要支持“为当前阶段补选必需任务”。

运行态需区分两种场景：

1. 可选切换目标
2. 必须先选任务才能进入当前阶段

第二种场景必须是阻塞态，不能绕过。

## 8. 异常与边界情况

### 8.1 旧数据兼容

旧的 phase 数据没有 `timer_mode` 和 `task_id` 时：

1. `focus` 阶段默认按 `countdown`
2. `task_id` 视为 `null`
3. 旧预设无需迁移脚本即可读取

### 8.2 已失效任务绑定

如果预设或活动工作流中的阶段绑定任务已不存在、已删除、或已完成：

1. 不得直接报 500
2. 应按“无效绑定”处理
3. 在可执行判断时跳过或进入待选任务状态

### 8.3 多来源任务完成

任务可能通过多种入口完成：

1. 主任务列表手动勾选完成
2. 快捷操作“完成并停止专注”
3. 任务编辑弹窗修改状态

这些路径都必须统一触发活动工作流重评估，避免出现 UI 已完成但工作流仍停在旧任务上的状态。

### 8.4 正计时异常累计保护

`countup` 阶段必须保证：

1. 页面刷新后仍能根据 `phase_started_at` 恢复已用时
2. 即便前端掉线，后端也能在查询当前工作流时识别并截断到 `2h`
3. 任何超出 `2h` 的专注记录都不得继续累积

## 9. 测试设计

### 9.1 后端测试

重点增加：

1. 预设 phase 新字段的校验与序列化测试
2. `countup` 阶段达到目标时长但未到 `2h` 的状态测试
3. `countup` 阶段到 `2h` 自动结束测试
4. 当前阶段任务完成后自动推进测试
5. 后续多个已完成任务阶段被连续跳过测试
6. 跳过无意义 `break` 的扫描测试
7. 无绑定任务阶段进入 `pending_task_selection` 测试
8. 运行时选任务接口测试

### 9.2 前端测试

重点增加：

1. 预设编辑界面中每阶段任务绑定和计时模式配置的表单行为测试
2. `countdown` 与 `countup` 两种阶段的展示测试
3. `pending_task_selection` 状态的阻塞 UI 测试
4. 当前阶段任务完成后 UI 自动切换测试
5. 自动跳过多个无效阶段后的显示正确性测试

## 10. 实现建议顺序

1. 先扩展后端 phase 模型与预设校验。
2. 再实现活动工作流运行态字段和统一推进函数。
3. 然后补运行时选任务接口。
4. 最后改前端预设编辑与运行态 UI。
5. 完成后补齐后端与前端回归测试。
