# 开发日志按分支分组设计

## 背景

当前 `docs/开发日志.md` 采用“按时间建标题”的结构。随着同一分支出现多次提交，日志中会重复出现分支信息，阅读时需要在整篇文档中来回查找，不利于追踪某个分支的连续开发过程。

仓库已经约定：每次代码提交后，需要记录工作分支、提交哈希、完成工作和验证情况。因此，开发日志需要一种更稳定、可持续追加的组织方式。

## 目标

- 将开发日志的主组织维度从“时间”切换为“分支”
- 保证每个分支只出现一次主分组
- 同一分支下按提交维度持续追加记录
- 保留现有日志要求：提交信息、完成工作、验证情况
- 保持长期维护成本低，避免频繁重排整篇文档

## 非目标

- 不引入自动生成脚本
- 不为开发日志增加额外索引页或统计页
- 不改变“每次提交后需要记录日志”的仓库约定
- 不改变单条提交记录的核心字段语义

## 方案对比

### 方案一：分支固定分组

以分支作为二级标题，每个分支只保留一个分组。新的提交记录写入对应分支分组顶部，分支分组本身保持首次出现时的位置不变。

优点：

- 结构稳定，阅读路径清晰
- 同一分支历史天然聚合
- 后续补录或回到老分支继续开发时，不会打乱整篇文档顺序
- diff 更可控，适合长期维护

缺点：

- 最近活跃的老分支不一定出现在全文最前面

### 方案二：分支按最近活跃重排

同样按分支分组，但只要某个老分支出现新提交，就把整个分支块移动到文档顶部。

优点：

- 最近活跃内容最显眼

缺点：

- 文档位置频繁变化
- 同一分支块会反复移动，历史定位不稳定
- 对日志这种长期累计文档不友好

### 方案三：顶部索引 + 分支固定分组

正文采用方案一，同时在文档顶部维护“最近活跃分支”索引。

优点：

- 同时兼顾稳定性与查找速度

缺点：

- 维护成本更高
- 对当前规模而言收益有限

## 选型结论

采用方案一：分支固定分组。

原因：

- 符合当前最重要的诉求：按分支追踪连续开发过程
- 能处理“过几天回到老分支继续开发”的场景，且不会生成重复分组
- 不需要额外索引和重排规则，维护心智负担最小

## 目标文档结构

`docs/开发日志.md` 调整为以下层次：

- 一级标题：`# 开发日志`
- 规则说明区：描述记录方式和维护要求
- 二级标题：分支名，例如 `## codex/repeat-completion-events`
- 三级标题：单次提交，例如 `### 2026-04-20 14:46 · 44fed46 · fix(todo): keep manually scheduled tasks in today summary`

每条提交记录下保留以下内容：

- 本次完成的工作
- 验证情况
- 可选备注

## 记录规则

### 分支分组规则

- 新分支第一次出现时，新建一个分支分组
- 已存在的分支再次提交时，继续写入原分支分组
- 不允许为同一分支创建多个分组
- 分支分组保持首次出现顺序，不因后续提交而整体重排

### 分支内排序规则

- 同一分支下，最新提交放在最上方
- 历史提交按时间倒序排列
- 如果后续需要补录旧提交，也应插入到正确时间位置

### 单条记录规则

每条提交记录至少包含：

- 提交时间
- 提交哈希
- 提交信息
- 本次完成的工作
- 验证情况

可选包含：

- 备注

### 特殊情况规则

- 文档类提交也可以记录；若无代码验证，则明确写“文档提交，无额外代码验证”
- 若该次提交未执行测试，也必须明确写“未验证”或对应原因，不能留空
- 工作区存在本地未跟踪文件时，不应自动纳入日志内容

## 对仓库规则的影响

`AGENTS.md` 中关于开发日志的要求应与新结构保持一致。仓库规则仍然强调“每次代码提交后更新开发日志”，但表述应从“追加一条时间记录”转为“在对应分支分组下新增一条提交记录”。

## 实施步骤

1. 调整 `docs/开发日志.md` 的维护规则说明，改为按分支分组
2. 将现有提交记录按分支归并到对应分组
3. 将同一分支下的提交整理为倒序列表
4. 同步更新 `AGENTS.md` 中与开发日志维护方式相关的表述

## 验收标准

- `docs/开发日志.md` 中每个分支只出现一个分组标题
- 同一分支的多次提交集中展示在同一分组下
- 同一分支内新提交位于旧提交之前
- 回到旧分支补记提交时，不需要新增同名分组
- `AGENTS.md` 与开发日志实际结构不矛盾
