# 开发日志补充用户需求原话设计

## 背景

当前 `docs/开发日志.md` 已按分支分组维护，并要求每次代码提交后记录提交信息、本次完成的工作和验证情况。这种结构能追踪“做了什么”，但还不能直接回答“这次提交是在响应用户的哪条需求”。

在实际协作中，开发日志除了记录实现结果，也需要回溯需求来源。用户要求每次提交不仅有工作记录，还要记录用户提出的需求，并且希望尽量保留用户原话的表达，而不是只写抽象摘要。

此外，这一要求不仅适用于未来的新提交，也要回补到已经存在的历史提交中。

## 目标

- 为每条提交记录新增固定字段 `用户提出的需求`
- 内容优先保留用户原话，只做最小整理
- 历史提交和未来新提交都遵循相同结构
- 保持日志可读性，不把需求来源和实现结果混写
- 同步更新仓库规则，确保后续提交持续执行

## 非目标

- 不要求逐字转录整段对话
- 不引入“需求编号”“会话链接”“任务编号”等额外追踪系统
- 不自动从聊天记录生成日志内容
- 不改变“按分支分组、分支不重排”的既有结构

## 方案对比

### 方案一：单独区块记录用户原话

在每条提交记录中新增一个固定区块：

- 用户提出的需求
- 本次完成的工作
- 验证情况

其中 `用户提出的需求` 下可包含 1 到多条 bullet，优先摘录最直接驱动该提交的用户原话。

优点：

- 需求来源和实现结果分层清晰
- 最符合“尽量保留用户原话”的诉求
- 回看历史时能快速理解每次提交的触发原因

缺点：

- 单条日志会更长

### 方案二：把需求揉进“本次完成的工作”

不新增字段，只在“本次完成的工作”前增加一两句“响应了用户提出的……”。

优点：

- 结构最短

缺点：

- “用户想要什么”和“我们做了什么”混在一起
- 后续快速扫读日志时，不容易分辨需求与实现

### 方案三：把需求写进提交标题

将三级标题改成 `时间 · 哈希 · 提交信息 · 需求摘要`。

优点：

- 需求一眼可见

缺点：

- 标题会过长
- 不适合保留用户原话
- 一个提交如果对应多条需求，标题会变得混乱

## 选型结论

采用方案一：单独区块记录用户原话。

原因：

- 兼顾可读性和可追溯性
- 能在不破坏现有结构的前提下平滑扩展
- 最容易落实“尽量保留用户原话、仅做最小整理”的原则

## 目标文档结构

每条提交记录调整为以下固定结构：

```md
### 2026-04-20 14:46 · 44fed46 · fix(todo): keep manually scheduled tasks in today summary

- 用户提出的需求：
  - “我想在安排管理右边的任务板块中仍然可以看到已完成的任务，但是无法拖到左边的今日任务中”
  - “编辑任务是不是还没有更新，我看到很多旧参数”

- 本次完成的工作：
  - 调整安排总览“今日任务”的最终展示规则
  - ...

- 验证情况：
  - `uv run pytest tests/test_todo_routes.py`
  - `pnpm check`
```

提交标题本身不变，新增字段只出现在标题下方。

## 记录规则

### 用户需求字段规则

- 每条提交记录必须包含 `用户提出的需求`
- 该字段优先使用用户原话
- 允许对原话进行最小整理，包括：
  - 删除不影响语义的口头语
  - 将连续表达拆分为多条 bullet
  - 对明显重复内容做去重
- 不允许把原本明确的用户要求改写成过度抽象的总结

### 原话选取规则

- 优先选取最直接驱动该提交的 1 到 3 条用户表达
- 如果一次提交由多轮连续讨论推动，可记录多条原话
- 如果某条原话过长，可以摘取其中最关键的句子，但应保持原意和语气
- 如果提交是文档或流程类调整，也要记录对应的用户要求原话

### 历史补写规则

- 已存在的历史提交也要补写 `用户提出的需求`
- 历史补写时，以当前已知对话记录为依据
- 如果同一提交可对应多条候选原话，应优先选择与提交信息最贴近的表达
- 如果无法完全一一映射，也应选择最能代表该次提交动机的原话，而不是留空

### 未来维护规则

- 后续每次代码提交后，在对应分支分组下新增提交记录时，同步填写：
  - 用户提出的需求
  - 本次完成的工作
  - 验证情况
- 若某次提交没有执行测试，也必须在验证情况中明确说明

## 对仓库规则的影响

`AGENTS.md` 中关于开发日志的维护要求应同步更新，从“记录提交哈希、完成工作、验证情况”扩展为“记录用户提出的需求、完成工作、验证情况”。

## 实施步骤

1. 更新 `docs/开发日志.md` 的维护规则，加入 `用户提出的需求` 必填要求
2. 为现有历史提交补写 `用户提出的需求` 区块
3. 调整后续日志模板，确保新提交沿用同一结构
4. 更新 `AGENTS.md` 中关于开发日志记录字段的说明

## 验收标准

- `docs/开发日志.md` 中每条提交记录都包含 `用户提出的需求`
- 该字段内容以用户原话为主，而不是抽象标题化总结
- 历史提交已补写，不存在旧记录缺失该字段
- `AGENTS.md` 的规则与日志结构一致
