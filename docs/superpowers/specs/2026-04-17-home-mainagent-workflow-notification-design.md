# 首页 MainAgent 工作流阶段提醒设计规格 (Spec)

## 1. 目标与背景

当前首页已经存在一个 `MainAgent` 自动开场能力：当用户进入或刷新首页时，首页对话区会通过现有 websocket run 机制触发一次 AI 主动开场。这说明首页 AI 已经具备“在特定页面事件下主动说一句话”的基础能力。

与此同时，专注工作流已经具备明确的阶段状态与切换逻辑，包括：

1. `focus -> break`
2. `break -> focus`
3. `pending_confirmation`
4. `pending_task_selection`
5. 工作流结束并回到 `normal`

当前这些变化主要通过工作流 UI 自身表达，首页 `MainAgent` 不会在阶段切换时主动提醒用户。用户如果没有盯着工作流胶囊或弹窗，容易错过“该休息了”“需要确认继续”“需要先选任务”这类关键动作。

本次设计的目标是：

1. 当用户位于首页时，让 `MainAgent` 在工作流关键阶段变化后主动提醒用户。
2. 复用现有首页自动开场的“前端检测事件 -> 触发 websocket run -> 生成真实 assistant_message”链路。
3. 让提醒消息进入真实首页会话，而不是本地伪造一条 UI 消息。
4. 保持提醒简短、稳定、克制，避免把阶段提醒做成打断型闲聊。

## 2. 非目标

本次不做以下事情：

1. 不改动后端工作流状态机、阶段推进规则或工作流数据库结构。
2. 不把工作流提醒改成系统通知、浏览器通知或全局 toast。
3. 不引入新的后端“发消息到首页 AI”的专用接口。
4. 不让提醒消息代替首页自动开场。
5. 不在非首页路由触发这类提醒，也不离线补发用户离开首页期间错过的提醒。

## 3. 现状梳理

### 3.1 首页 MainAgent 自动开场

当前首页 `MainAgent` 自动开场逻辑位于 [`frontend/src/hooks/useAgentChat.ts`](/Users/qishu/Project/ark/frontend/src/hooks/useAgentChat.ts)。

其关键特征：

1. 仅在 `profileKey === 'MainAgent'` 且当前 hash 路由为 `/` 时触发。
2. 通过 `sendMessage(buildAutoOpenPrompt())` 直接走现有 websocket `run` 机制。
3. 触发后生成真实的 `assistant_message`，会进入首页现有消息流。
4. 具备冷却时间，避免每次刷新都重复打招呼。

这条链路已经证明：前端在满足特定页面条件时，触发一次“系统来源的 AI 主动发话”是可行的。

### 3.2 首页工作流状态获取

首页工作流状态目前主要通过 [`frontend/src/components/PlaceholderCard.tsx`](/Users/qishu/Project/ark/frontend/src/components/PlaceholderCard.tsx) 拉取并维护：

1. 初始化加载当前工作流快照。
2. 监听 `ark:reload-focus` 重新拉取。
3. 通过定时刷新保持工作流时间同步。

因此，首页已经拥有“当前工作流状态”的前端权威视图，适合作为阶段变化的检测位置。

## 4. 方案选择

本次采用“前端检测工作流关键事件，再触发首页 `MainAgent` 自己运行一次”的方案。

即：

1. 首页前端对比工作流前后快照。
2. 检测到关键阶段事件后，构造一段受控 prompt。
3. 通过 `MainAgent` 现有 websocket run 链路触发一次系统消息。
4. 由 `MainAgent` 产出真实 assistant 消息进入对话区。

不采用其他方案的原因：

1. 不直接向 UI 注入假消息，避免与真实会话历史分裂。
2. 不把工作流模块和后端 session/message 系统直接耦合，控制本次改动范围。

## 5. 关键体验设计

### 5.1 消息风格

工作流提醒不是新的开场，也不是聊天建议。

因此它必须满足以下约束：

1. 每次只围绕当前工作流事件本身。
2. 用 `1-2` 句短提醒。
3. 不寒暄，不重新自我介绍。
4. 不扩展成长篇说明。

### 5.2 消息内容策略

采用“事件固定、措辞轻微变化”的策略。

含义是：

1. 同一类事件的核心信息固定。
2. 措辞允许轻微变化，避免每次完全机械复读。
3. 若存在任务名或工作流名，可在消息中自然带上。

### 5.3 覆盖事件

本次提醒覆盖以下事件：

1. `focus -> break`
2. `break -> focus`
3. 进入 `pending_confirmation`
4. 进入 `pending_task_selection`
5. 工作流完成并回到 `normal`

### 5.4 各事件语义

#### `focus -> break`

核心信息：

1. 当前专注段结束。
2. 现在进入休息阶段。

示例语气：

1. “这一轮专注结束了，现在进入休息阶段。”
2. “你刚完成当前专注段，可以先休息一下。”

#### `break -> focus`

核心信息：

1. 当前休息段结束。
2. 现在回到专注阶段。

示例语气：

1. “休息时间到了，接下来回到专注阶段。”
2. “这一段休息结束了，可以继续专注了。”

#### `pending_confirmation`

核心信息：

1. 当前阶段已经结束。
2. 需要用户确认后才能继续流转。

示例语气：

1. “当前阶段已经结束，等你确认后我再陪你切到下一段。”
2. “这一段先停在这里，确认后就能继续流转。”

#### `pending_task_selection`

核心信息：

1. 下一个专注阶段还没有选定任务。
2. 需要先选择任务才能继续。

示例语气：

1. “下一个专注阶段还没选任务，先挑一个继续吧。”
2. “接下来要进入专注，但还需要你先选定任务。”

#### 工作流完成

核心信息：

1. 当前工作流已经结束。
2. 可以自然收尾，不再等待下一个阶段。

示例语气：

1. “这个工作流已经完成了。”
2. “这一套流程走完了，当前工作流已经结束。”

## 6. 触发条件与边界

### 6.1 页面条件

提醒仅在以下条件下允许触发：

1. 当前页面是首页 `/`
2. 首页 `MainAgent` 已挂载
3. `MainAgent` websocket 已可发送消息

如果用户不在首页，则不触发提醒，也不补发离开期间的提醒。

### 6.2 工作流快照对比原则

提醒不是基于时间流逝，而是基于“工作流状态语义变化”。

因此：

1. 普通倒计时变化不算事件。
2. `remaining_seconds` 每秒递减不应触发提醒。
3. 只有状态、阶段索引、待确认标志、待选任务标志、完成状态发生关键变化时才触发。

### 6.3 事件识别规则

设 `prev` 为上一份工作流快照，`next` 为当前快照。

触发规则如下：

1. 若 `prev.state === 'focus'` 且 `next.state === 'break'`，触发“进入休息”提醒。
2. 若 `prev.state === 'break'` 且 `next.state === 'focus'`，触发“回到专注”提醒。
3. 若 `prev.pending_confirmation !== true` 且 `next.pending_confirmation === true`，触发“等待确认继续”提醒。
4. 若 `prev.pending_task_selection !== true` 且 `next.pending_task_selection === true`，触发“等待选择任务”提醒。
5. 若 `prev.state !== 'normal'` 且 `next.state === 'normal'`，且存在工作流完成上下文，触发“工作流完成”提醒。

### 6.4 优先级

单次快照变化只触发一种提醒，避免一次状态刷新产生多句消息。

推荐优先级如下：

1. 工作流完成
2. `pending_task_selection`
3. `pending_confirmation`
4. `focus -> break`
5. `break -> focus`

原因：

1. 阻塞态和完成态对用户动作更关键。
2. 若一个状态变化同时反映阶段切换和阻塞语义，应优先提醒用户接下来需要做的事。

## 7. 去重设计

### 7.1 去重目标

首页当前会因为多种原因重复获取同一份工作流状态：

1. 定时刷新
2. `ark:reload-focus`
3. 组件重新挂载
4. 页面刷新恢复

如果不去重，同一个事件可能让 `MainAgent` 重复说多次。

### 7.2 去重键

前端为每个可提醒事件生成一个稳定的事件 key。

key 应至少包含：

1. 事件类型
2. 工作流名
3. 当前阶段索引（如适用）
4. 当前任务 id 或运行任务 id（如适用）
5. 完成标识（工作流完成时）

示意：

1. `workflow:focus_to_break:写作流:1:task-123`
2. `workflow:pending_confirmation:默认工作流:2:task-123`
3. `workflow:completed:默认工作流`

### 7.3 去重行为

首页只缓存最近一次成功触发发送的事件 key。

规则：

1. 若新计算出的事件 key 与最近一次已发送 key 相同，则不再触发。
2. 若事件 key 变化，则允许新的提醒触发。
3. 页面刷新后允许重新开始一轮去重，但同一次状态恢复不应在初始化过程中连续触发多次。

本次设计默认以内存级去重为主，不要求跨刷新持久化工作流提醒历史。

## 8. MainAgent 集成设计

### 8.1 总体方式

复用 [`frontend/src/hooks/useAgentChat.ts`](/Users/qishu/Project/ark/frontend/src/hooks/useAgentChat.ts) 中现有的 websocket run 机制。

不新增新的消息注入 API。

### 8.2 抽象目标

将“系统来源主动触发一条 `MainAgent` 消息”的能力从首页自动开场逻辑中抽象出来，供工作流提醒复用。

抽象后的能力应支持：

1. 指定来源标识
2. 传入受控 prompt
3. 使用现有 `sendMessage(...)` 发送
4. 保持与当前 `assistant_message` 流一致

### 8.3 与自动开场的关系

首页自动开场与工作流提醒将共用同一个“系统触发 run”能力，但语义上是两类不同事件。

它们的区别：

1. 自动开场来源于“页面进入/刷新首页”
2. 工作流提醒来源于“工作流阶段语义变化”

因此：

1. 两者需要不同的来源标识
2. 两者应分别管理自己的去重或冷却逻辑
3. 工作流提醒不应复用自动开场的一小时冷却时间

## 9. Prompt 设计

### 9.1 Prompt 内容

工作流提醒 prompt 应包含：

1. 来源标识
2. 当前事件类型
3. 当前工作流名
4. 当前任务名（如果有）
5. 当前状态上下文
6. 输出限制

### 9.2 Prompt 约束

prompt 必须强约束模型行为：

1. 只输出 `1-2` 句。
2. 不重新打招呼。
3. 不说“第一次见面”。
4. 若历史消息中有与本条冲突的格式要求，以本条规则为准。
5. `event.type` 优先于 `state` 字段语义，例如 `pending_confirmation` 必须提醒“等待确认继续”。
6. 不要扩展为闲聊。
7. 若适合当前事件，可以追加最多 `2` 个 `<suggestions>` 项。
8. 只围绕当前工作流事件做短提醒。

### 9.3 为什么仍使用 AI 而不是纯模板

本次仍使用 AI 输出，而不是完全本地模板，原因是：

1. 与首页 `MainAgent` 现有行为保持一致。
2. 可以自然带入工作流名和任务名。
3. 可以在固定语义范围内产生轻微措辞变化。

但本次通过强约束 prompt 将变化范围收紧，避免生成发散。

2. 是否满足首页限定条件
3. 是否正确去重

## 11. 风险与规避

### 11.1 重复提醒风险

风险：

1. 同一事件在轮询与 reload 中被多次识别。

规避：

1. 使用稳定事件 key。
2. 仅对语义变化做检测，不对时间变化做检测。

### 11.2 自动开场与工作流提醒冲突

风险：

1. 用户刷新首页时，自动开场与工作流提醒同时触发，导致连续两条系统来源消息。

规避：

1. 保持两类来源分开。
2. 实现时在首页初始化阶段控制先后顺序，避免同一渲染周期内无节制连发。
3. 若工作流状态在初始恢复时只是“已有阻塞态”而非“刚发生变化”，不应误判为新的阶段提醒。

### 11.3 模型输出跑偏

风险：

1. AI 输出冗长或闲聊。

规避：

1. 使用强约束 prompt。
2. 将提醒定义为短通知，而非会话开场。

## 12. 实现范围总结

本次实现预计只涉及前端：

1. 首页工作流状态变化检测与去重
2. `useAgentChat` 中系统触发 run 能力抽象
3. 工作流提醒 prompt 构造
4. 对应测试补充

后端工作流 API 与 session/message 持久化模型不在本次改动范围内。
